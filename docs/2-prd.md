## 1. 프로젝트 개요

### 1.1 프로젝트명
**Jan TodoList** - 개인 할일 관리 웹 애플리케이션

### 1.2 프로젝트 배경
일상 생활에서 할일을 효율적으로 관리하고, 기한을 놓치지 않도록 돕는 간단하면서도 직관적인 TODO 앱이 필요합니다.

### 1.3 개발 기간
**3일 스프린트** (2025-11-26 ~ 2025-11-28)

### 1.4 핵심 가치 제안
- ✅ **간단함**: 복잡한 설정 없이 바로 사용 가능
- 📅 **기한 관리**: 기한 기반 자동 정렬로 중요한 할일을 놓치지 않음
- 📱 **어디서나**: 모바일, 태블릿, 데스크톱 모두 지원
- 🔒 **개인화**: 회원 가입으로 어디서든 내 할일에 접근

## 2. 제품 목표 및 비전

### 2.1 제품 목표 (3일 스프린트)
1. **핵심 기능 구현**: 회원가입, 로그인, 할일 CRUD, 기한 관리
2. **완성도 있는 UI/UX**: 단순 프로토타입이 아닌 실사용 가능한 수준의 디자인
3. **프로덕션 배포**: 실제 URL로 접근 가능한 서비스 런칭

### 2.2 장기 비전 (Phase 2 이후)
- 알림 시스템 (기한 임박 알림)
- 캘린더 뷰 (월간 달력 형태)
- 공유 기능 (팀 협업)
- 외부 캘린더 연동 (구글/네이버)

### 2.3 성공 정의
3일 후:
- ✅ 친구/가족에게 URL을 공유하고 실제 사용 가능
- ✅ 모바일과 데스크톱 모두에서 원활하게 동작
- ✅ 버그 없이 안정적으로 작동

## 3. 대상 사용자

### 3.1 Primary Persona: 개인 사용자 (일반)

**이름**: 김민준 (27세)
**직업**: 대학원생 / 프리랜서
**기술 수준**: 중급 (스마트폰, 웹 서비스 익숙)
**사용 맥락**: 일상 생활 할일 관리

#### 페인 포인트
- 📝 "할일을 메모장에 적어두면 어디 적었는지 찾기 힘들어요"
- ⏰ "중요한 마감일을 자주 까먹어서 문제가 생겨요"
- 📱 "모바일에서 빠르게 할일을 추가하고 싶은데 앱을 설치하기는 귀찮아요"
- 🔄 "여러 기기에서 동기화되는 TODO 앱이 필요해요"

#### 사용 시나리오
1. **아침**: 침대에서 스마트폰으로 오늘 할일 확인
2. **낮**: 카페에서 노트북으로 새로운 할일 추가
3. **저녁**: 완료한 할일 체크, 내일 할일 정리

#### 목표
- 모든 할일을 한 곳에서 관리
- 기한이 임박한 할일을 놓치지 않기
- 빠르고 간편하게 할일 추가/완료 처리

### 3.2 사용 빈도
- **일 1-3회** 접속
- **회당 3-10분** 사용
- **주 10-20개** 할일 생성

## 4. 핵심 기능 요구사항

### 4.1 인증 및 사용자 관리 [REQ-001, REQ-002]

#### 4.1.1 회원가입
**사용자 스토리**:
> "개인 사용자로서, 이메일과 비밀번호로 간단하게 회원가입하고 싶다, 그래서 어디서든 내 할일에 접근할 수 있도록"

**기능 상세**:
- 이메일, 비밀번호, 이름 입력
- 이메일 형식 검증 (클라이언트 + 서버)
- 비밀번호 최소 8자 이상 검증
- 중복 이메일 체크
- bcrypt 해싱 (최소 10 rounds)
- 성공 시 자동 로그인

**인수 기준**:
- [ ] 유효한 이메일 형식 검증 (예: test@example.com)
- [ ] 비밀번호 8자 미만 시 오류 메시지
- [ ] 중복 이메일 시 "이미 사용 중인 이메일입니다" 메시지
- [ ] 회원가입 성공 시 자동 로그인 후 할일 목록 화면으로 이동
- [ ] 서버 응답 시간 2초 이내

**UI/UX**:
```
┌─────────────────────────┐
│   Jan TodoList          │
│   회원가입              │
│                         │
│  이메일                 │
│  [________________]     │
│                         │
│  비밀번호               │
│  [________________]     │
│                         │
│  이름                   │
│  [________________]     │
│                         │
│  [    가입하기    ]     │
│                         │
│  이미 계정이 있나요?    │
│  로그인 →               │
└─────────────────────────┘
```

#### 4.1.2 로그인
**사용자 스토리**:
> "개인 사용자로서, 이메일과 비밀번호로 로그인하고 싶다, 그래서 내 할일 목록을 확인할 수 있도록"

**기능 상세**:
- 이메일, 비밀번호 입력
- JWT 토큰 발급 (만료 시간: 24시간)
- localStorage에 토큰 저장
- 자동 로그인 유지 (토큰 유효 시)

**인수 기준**:
- [ ] 올바른 이메일/비밀번호로 로그인 성공
- [ ] 잘못된 이메일/비밀번호 시 "이메일 또는 비밀번호가 일치하지 않습니다" 메시지
- [ ] 로그인 성공 시 할일 목록 화면으로 이동
- [ ] 토큰 만료 시 자동 로그아웃 및 로그인 화면으로 리다이렉트
- [ ] 서버 응답 시간 2초 이내

#### 4.1.3 로그아웃
**기능 상세**:
- localStorage에서 토큰 삭제
- 로그인 화면으로 리다이렉트

**인수 기준**:
- [ ] 로그아웃 버튼 클릭 시 즉시 로그아웃
- [ ] 로그인 화면으로 자동 이동
- [ ] 뒤로 가기로 할일 목록 접근 불가

### 4.2 할일 관리 (CRUD) [REQ-003, REQ-005]

#### 4.2.1 할일 생성
**사용자 스토리**:
> "개인 사용자로서, 새로운 할일을 빠르게 추가하고 싶다, 그래서 잊어버리지 않고 관리할 수 있도록"

**기능 상세**:
- 제목 입력 (필수, 1-200자)
- 기한 설정 (선택, 날짜만, YYYY-MM-DD)
- 기본 상태: 미완료 (is_completed = false)
- 생성일시 자동 기록

**인수 기준**:
- [ ] 제목만으로 할일 생성 가능 (기한 없이)
- [ ] 제목 1자 미만 시 "제목을 입력해주세요" 메시지
- [ ] 제목 200자 초과 시 입력 제한
- [ ] 과거 날짜 기한 설정 시 "오늘 이후 날짜를 선택해주세요" 메시지
- [ ] 생성 성공 시 목록 최상단 (또는 기한 순서에 맞게) 추가
- [ ] 응답 시간 500ms 이내

**UI/UX**:
- 모바일: 하단 고정 "+ 할일 추가" 버튼
- 데스크톱: 상단 입력 폼
- 빠른 추가: Enter 키로 즉시 생성

#### 4.2.2 할일 조회 (목록)
**사용자 스토리**:
> "개인 사용자로서, 내 모든 할일을 한눈에 보고 싶다, 그래서 무엇을 해야 할지 빠르게 파악할 수 있도록"

**기능 상세**:
- 사용자별 할일 목록 조회
- 기한 기반 자동 정렬:
  1. 기한이 가까운 순 (ASC)
  2. 동일 기한: 생성일시 최신 순 (DESC)
  3. 기한 없음: 맨 아래
- 완료/미완료 상태 표시
- 기한 표시 (있는 경우)

**인수 기준**:
- [ ] 로그인한 사용자의 할일만 조회
- [ ] 기한 임박 할일이 상단에 표시
- [ ] 완료된 할일 시각적 구분 (예: 회색 처리, 취소선)
- [ ] 빈 목록 시 "할일을 추가해보세요!" 안내 메시지
- [ ] 응답 시간 1초 이내

**정렬 예시**:
```
1. [미완료] 프로젝트 제출 (기한: 11월 27일)  ← 내일
2. [미완료] 병원 예약 (기한: 11월 28일)    ← 모레
3. [완료] 장보기 (기한: 11월 26일)         ← 오늘 (완료됨)
4. [미완료] 책 읽기 (기한 없음)            ← 기한 없음
```

#### 4.2.3 할일 수정
**사용자 스토리**:
> "개인 사용자로서, 할일의 제목이나 기한을 수정하고 싶다, 그래서 변경된 상황을 반영할 수 있도록"

**기능 상세**:
- 제목 수정
- 기한 수정 (추가/변경/제거)
- 완료 상태 토글
- 소유자만 수정 가능

**인수 기준**:
- [ ] 할일 클릭 시 수정 모드 진입
- [ ] 제목/기한 수정 후 즉시 반영
- [ ] 완료 체크박스 클릭으로 상태 토글
- [ ] 다른 사용자의 할일 수정 불가 (서버 검증)
- [ ] 응답 시간 500ms 이내

**UI/UX**:
- 인라인 편집 (클릭 시 바로 수정 모드)
- ESC 키로 취소
- Enter 키 또는 포커스 아웃으로 저장

#### 4.2.4 할일 삭제
**사용자 스토리**:
> "개인 사용자로서, 불필요한 할일을 삭제하고 싶다, 그래서 목록을 깔끔하게 유지할 수 있도록"

**기능 상세**:
- 소유자만 삭제 가능
- 삭제 확인 대화상자
- 물리 삭제 (DB에서 완전 제거)

**인수 기준**:
- [ ] 삭제 버튼 클릭 시 확인 대화상자 표시
- [ ] "정말 삭제하시겠습니까?" 메시지
- [ ] 확인 시 즉시 삭제 및 목록에서 제거
- [ ] 취소 시 아무 변화 없음
- [ ] 응답 시간 500ms 이내

- 오늘 이후 날짜만 허용

- 선택 사항 (기한 없는 할일 허용)

**인수 기준**:

- [ ] 과거 날짜 선택 불가
- [ ] 기한 제거 옵션 ("기한 없음" 버튼)
- [ ] 기한 변경 시 자동 재정렬

#### 4.3.2 기한 기반 정렬
**비즈니스 로직**:
```javascript
정렬 우선순위:
1. 기한 있음 && 미완료 → 기한 가까운 순 (ASC)
2. 기한 있음 && 완료 → 기한 가까운 순 (ASC)
3. 기한 없음 && 미완료 → 생성일 최신 순 (DESC)
4. 기한 없음 && 완료 → 생성일 최신 순 (DESC)
```

**인수 기준**:
- [ ] 기한 임박 할일이 최상단
- [ ] 기한 없는 할일이 최하단
- [ ] 완료된 할일은 별도 구분 (회색 처리)
- [ ] 정렬 정확도 100%

#### 4.3.3 기한 시각적 표시
**UI/UX**:
```
기한 표시 방식:
- 오늘: "오늘" (빨간색)
- 내일: "내일" (주황색)
- 이번 주: "11월 28일 (목)" (노란색)
- 그 이후: "12월 1일" (회색)
- 기한 없음: 표시 안 함
```

**인수 기준**:
- [ ] 기한별 색상 구분 명확
- [ ] "오늘", "내일" 텍스트 자동 변환
- [ ] 모바일에서도 가독성 유지

## 5. UI/UX 요구사항

### 5.1 디자인 원칙
1. **미니멀**: 불필요한 요소 최소화, 할일에 집중
2. **직관적**: 설명 없이도 사용 가능한 인터페이스
3. **빠른 인터랙션**: 클릭 1-2번으로 모든 작업 완료
4. **일관성**: 모든 화면에서 동일한 디자인 언어

### 5.2 화면 구성

#### 5.2.1 필수 화면 (Phase 1)
1. **로그인 화면** (`/login`)
   - 이메일, 비밀번호 입력
   - "회원가입" 링크

2. **회원가입 화면** (`/signup`)
   - 이메일, 비밀번호, 이름 입력
   - "로그인" 링크

3. **할일 목록 화면** (`/todos` - 메인)
   - 헤더: 로고, 사용자 이름, 로그아웃 버튼
   - 할일 추가 입력 폼 (상단)
   - 할일 목록 (스크롤 가능)
   - 빈 목록 안내 메시지

4. **404 페이지** (`*`)
   - "페이지를 찾을 수 없습니다" 메시지
   - 홈으로 돌아가기 버튼

#### 5.2.2 제외 화면 (Phase 2)
- ❌ 설정 화면 (프로필 수정, 비밀번호 변경)
- ❌ 캘린더 뷰
- ❌ 공유 관리 화면

### 5.3 반응형 디자인

#### 5.3.1 브레이크포인트
```css
/* 모바일 */
@media (max-width: 767px) {
  - 세로 레이아웃
  - 하단 고정 추가 버튼
  - 풀 스크린 모달
}

/* 태블릿 */
@media (min-width: 768px) and (max-width: 1023px) {
  - 2단 레이아웃 가능
  - 사이드 패널
}

/* 데스크톱 */
@media (min-width: 1024px) {
  - 중앙 정렬 (최대 너비 1200px)
  - 좌우 여백
}
```

#### 5.3.2 모바일 최적화
- 터치 영역 최소 44x44px

### 6.4 데이터베이스 스키마

```sql
-- users 테이블
CREATE TABLE users (
  id SERIAL PRIMARY KEY,
  email VARCHAR(255) UNIQUE NOT NULL,
  password VARCHAR(255) NOT NULL,  -- bcrypt 해시
  name VARCHAR(100) NOT NULL,
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- todos 테이블
CREATE TABLE todos (
  id SERIAL PRIMARY KEY,
  user_id INTEGER NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  title VARCHAR(200) NOT NULL,
  is_completed BOOLEAN DEFAULT FALSE,
  due_date DATE,
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- 인덱스
CREATE INDEX idx_todos_user_id ON todos(user_id);
CREATE INDEX idx_todos_due_date ON todos(due_date);
```

8. 다른 사용자의 할일 삭제 → 실패 (403 Forbidden)

   ```

   // package.json
   {
     "scripts": {
       "start": "node src/server.js",
       "build": "echo 'No build step'"
     }
   }
   ```

#### 11.5.1 SQL Injection
**리스크**: 사용자 입력으로 SQL 공격

**완화 전략**:
- ✅ Prepared Statement 사용 (pg 라이브러리)
  ```javascript
  // Good
  await pool.query('SELECT * FROM users WHERE email = $1', [email]);

  // Bad (절대 사용 금지)
  await pool.query(`SELECT * FROM users WHERE email = '${email}'`);
  ```
- ✅ express-validator로 입력 검증
