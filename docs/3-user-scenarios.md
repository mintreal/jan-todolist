## 1. 개요

본 문서는 Jan TodoList의 Phase 1 범위(3일 스프린트)에 해당하는 사용자 시나리오를 상세히 기술합니다. 실제 사용자의 관점에서 서비스를 어떻게 이용하는지, 각 단계에서 어떤 인터랙션이 발생하는지를 구체적으로 설명합니다.

### Phase 1 기능 범위
- 회원가입/로그인
- 할일 CRUD (생성, 조회, 수정, 삭제)
- 기한 관리
- 반응형 UI

## 2. 페르소나

### 김민준 (27세, 대학원생/프리랜서)

**기본 정보**:
- 직업: 대학원 재학 중, 프리랜서 디자이너로 부업
- 기술 수준: 중급 (스마트폰, 웹 서비스 사용에 익숙)
- 사용 기기: iPhone 13, MacBook Pro

**일상**:
- 아침 8시 기상 → 침대에서 스마트폰으로 오늘 할일 확인
- 오전 9시 대학교 연구실 도착 → 논문 작업
- 오후 2시 카페 이동 → 프리랜서 프로젝트 작업
- 저녁 10시 귀가 → 완료한 할일 체크, 내일 계획 정리

**페인 포인트**:
- 논문 마감, 프로젝트 납품일 등 중요한 기한을 놓쳐서 스트레스 받음
- 메모장에 할일을 적어두면 어디 적었는지 찾기 힘듦
- 여러 기기를 사용하기 때문에 동기화가 안 되면 불편함
- 무거운 앱 설치는 싫고, 웹에서 빠르게 사용하고 싶음

**목표**:
- 모든 할일을 한 곳에서 관리
- 기한이 임박한 할일을 놓치지 않기
- 모바일과 노트북 모두에서 접근 가능

## 3. 기본 시나리오

### 시나리오 1: 첫 방문자의 회원가입 및 첫 할일 추가

**배경**:
민준이 친구 추천으로 Jan TodoList를 알게 되었습니다. 별도 앱 설치 없이 웹에서 바로 사용할 수 있다는 점이 마음에 들어 바로 시도해보기로 했습니다.

**목표**:
회원가입 후 첫 할일을 추가하여 서비스 사용을 시작한다.

**사전 조건**:
- 사용자는 Jan TodoList 계정이 없음
- Chrome 브라우저로 jan-todolist.com 접속

**단계**:

1. **랜딩 페이지 접속**
   - 행동: jan-todolist.com 접속
   - 화면: 로그인 화면 표시 (이메일, 비밀번호 입력란, "회원가입" 링크)
   - 예상 결과: 깔끔하고 간단한 UI, 서비스 로고 "Jan TodoList" 표시

2. **회원가입 화면 이동**
   - 행동: "회원가입" 링크 클릭
   - 화면 전환: 회원가입 화면 표시
   - 예상 결과: 부드러운 화면 전환 (0.2초 애니메이션), 이메일/비밀번호/이름 입력란

3. **회원 정보 입력**
   - 행동:
     - 이메일: minjun.kim@gmail.com 입력
     - 비밀번호: password1234 입력
     - 이름: 김민준 입력
   - 실시간 검증:
     - 이메일 형식 검증 (@ 포함 여부)
     - 비밀번호 8자 이상 검증 (하단에 "✓ 8자 이상" 표시)
   - 예상 결과: 각 입력란에 포커스 시 테두리 파란색 하이라이트

4. **가입하기 버튼 클릭**
   - 행동: "가입하기" 버튼 클릭
   - 서버 요청: POST /api/auth/signup
   - 로딩 표시: 버튼에 스피너 애니메이션 (1-2초)
   - 예상 결과:
     - 성공 시: JWT 토큰 발급, localStorage에 자동 저장
     - 자동 로그인 후 할일 목록 화면으로 리다이렉트

5. **할일 목록 화면 진입 (빈 상태)**
   - 화면 구성:
     - 헤더: "Jan TodoList" 로고, "김민준님" 표시, "로그아웃" 버튼
     - 중앙: "아직 할일이 없어요. 첫 할일을 추가해보세요!" 안내 메시지
     - 상단: 할일 추가 입력 폼 (제목, 기한)
   - 예상 결과: 깔끔한 빈 상태 UI, 입력 폼에 자동 포커스

6. **첫 할일 추가**
   - 행동:
     - 제목 입력: "논문 초고 작성"
     - 기한 선택: 오늘로부터 3일 후 (2025-11-29)
     - Enter 키 입력 또는 "추가" 버튼 클릭
   - 서버 요청: POST /api/todos
   - 예상 결과:
     - 할일이 목록 상단에 추가됨
     - 애니메이션: 위에서 아래로 슬라이드 인 (0.3초)
     - 입력 폼 자동 초기화
     - 성공 토스트 메시지: "할일이 추가되었습니다" (2초 후 사라짐)

7. **추가된 할일 확인**
   - 화면 표시:
     ```
     □ 논문 초고 작성
       11월 29일 (금) | 3일 후
     ```
   - 예상 결과:
     - 체크박스 (미완료 상태)
     - 제목 굵은 글씨
     - 기한 주황색 텍스트 ("3일 후")
     - 우측: 수정 아이콘, 삭제 아이콘

**완료 조건**:
- 회원가입 성공
- 자동 로그인
- 첫 할일이 목록에 표시됨
- 사용자가 서비스 사용 방법을 이해함

**대체 흐름**:

**3-1. 중복 이메일 오류**
- 조건: 이미 가입된 이메일 입력
- 결과: "이미 사용 중인 이메일입니다" 빨간색 오류 메시지
- 사용자 행동: 다른 이메일 입력 또는 "로그인" 링크 클릭

**3-2. 비밀번호 길이 부족**
- 조건: 7자 이하 비밀번호 입력
- 결과: 입력란 하단에 "비밀번호는 8자 이상이어야 합니다" 메시지
- 가입 버튼 비활성화 (회색 처리)

**6-1. 제목 없이 추가 시도**
- 조건: 제목 입력란이 비어있는 상태에서 Enter 또는 추가 버튼 클릭
- 결과: "제목을 입력해주세요" 메시지, 입력란에 빨간색 테두리

### 시나리오 2: 기존 사용자의 로그인 및 할일 관리

**배경**:
민준이 어제 회원가입을 마쳤고, 오늘 아침 다시 서비스에 접속했습니다.

**목표**:
로그인하여 할일 목록을 확인하고, 새 할일을 추가한다.

**사전 조건**:
- 회원가입 완료 (이메일: minjun.kim@gmail.com)
- 브라우저에 토큰이 없는 상태 (로그아웃 후 또는 쿠키 삭제)

**단계**:

1. **로그인 화면 접속**
   - 행동: jan-todolist.com 접속
   - 화면: 로그인 화면
   - 예상 결과: 이메일/비밀번호 입력란 표시

2. **로그인 정보 입력**
   - 행동:
     - 이메일: minjun.kim@gmail.com
     - 비밀번호: password1234
     - "로그인" 버튼 클릭
   - 서버 요청: POST /api/auth/login
   - 예상 결과:
     - JWT 토큰 발급
     - localStorage에 저장
     - 할일 목록 화면으로 리다이렉트 (1초 이내)

3. **할일 목록 확인**
   - 서버 요청: GET /api/todos
   - 화면 표시: 기존 할일 목록 (기한 순 정렬)
   - 예상 결과:
     ```
     □ 프로젝트 제출 (오늘, 빨간색)
     □ 논문 초고 작성 (내일, 주황색)
     □ 병원 예약 (11월 29일, 노란색)
     □ 책 읽기 (기한 없음, 회색)
     ```

4. **새 할일 추가**
   - 행동:
     - 제목: "장보기"
     - 기한: 오늘
     - Enter 키
   - 예상 결과:
     - 할일이 목록 최상단에 추가 (기한: 오늘이 최우선)
     - 자동 정렬

5. **할일 완료 처리**
   - 행동: "장보기" 체크박스 클릭
   - 서버 요청: PUT /api/todos/:id (is_completed: true)
   - 예상 결과:
     - 체크박스 체크됨 (✓)
     - 제목에 취소선 (~~장보기~~)
     - 회색 처리
     - 체크 애니메이션 (0.2초)

**완료 조건**:
- 로그인 성공
- 할일 목록 정상 표시
- 새 할일 추가 및 완료 처리 성공

**대체 흐름**:

**2-1. 잘못된 비밀번호**
- 조건: 올바르지 않은 비밀번호 입력
- 결과: "이메일 또는 비밀번호가 일치하지 않습니다" 오류 메시지
- 비밀번호 입력란 자동 초기화 및 포커스

**2-2. 네트워크 오류**
- 조건: 서버 연결 실패
- 결과: "네트워크 오류가 발생했습니다. 다시 시도해주세요" 메시지
- "재시도" 버튼 표시

## 4. 일상 사용 시나리오

### 시나리오 3: 아침 루틴 - 오늘 할일 확인 (모바일)

**배경**:
아침 8시, 민준이 침대에서 눈을 뜨자마자 iPhone으로 오늘 할일을 확인합니다.

**목표**:
모바일에서 빠르게 오늘 할 일을 확인하고 우선순위를 파악한다.

**사전 조건**:
- iPhone Safari 브라우저
- 로그인 상태 (토큰 유효)

**단계**:

1. **앱 접속**
   - 행동: Safari 북마크에서 Jan TodoList 클릭
   - 화면: 자동 로그인 (토큰 유효)
   - 로딩: 스켈레톤 UI (0.5초), 할일 목록 표시
   - 예상 결과: 모바일 최적화 레이아웃 (세로 스크롤)

2. **오늘 기한 할일 확인**
   - 화면 상단:
     ```
     □ 프로젝트 제출 (오늘, 빨간색, 큰 글씨)
     □ 교수님 미팅 준비 (오늘, 빨간색)
     ```
   - 예상 결과:
     - 오늘 기한 할일이 최상단
     - 빨간색 "오늘" 배지
     - 터치 영역 충분 (최소 44px)

3. **할일 빠르게 추가**
   - 행동:
     - 하단 고정 "+" 버튼 탭
     - 입력 모달 팝업
     - 제목: "우유 사기"
     - 기한: 오늘
     - "추가" 버튼 탭
   - 예상 결과:
     - 풀 스크린 모달 (모바일)
     - 키보드 자동 올라옴
     - 추가 후 모달 닫힘, 목록 최상단에 추가

4. **Pull-to-Refresh**
   - 행동: 화면 상단에서 아래로 당기기 (스와이프)
   - 예상 결과:
     - 리프레시 스피너 표시
     - GET /api/todos 재요청
     - 최신 목록 업데이트

**완료 조건**:
- 모바일에서 원활한 조회
- 빠른 할일 추가 성공
- 오늘 기한 할일 시각적으로 명확히 구분

**대체 흐름**:

**2-1. 오늘 기한 할일 없음**
- 화면: "오늘 기한인 할일이 없어요 👍" 안내 메시지
- "내일" 할일부터 표시

### 시나리오 4: 점심시간 - 카페에서 할일 수정 (데스크톱)

**배경**:
오후 2시, 민준이 카페에서 MacBook으로 프리랜서 작업 중, 할일 목록을 정리하려 합니다.

**목표**:
할일의 제목과 기한을 수정하고, 완료된 할일을 삭제한다.

**사전 조건**:
- MacBook Chrome 브라우저
- 로그인 상태

**단계**:

1. **할일 목록 조회 (데스크톱 레이아웃)**
   - 화면:
     - 중앙 정렬 (최대 너비 1200px)
     - 좌우 여백
     - 할일 추가 폼 상단 고정
     - 할일 목록 스크롤 가능
   - 예상 결과: 넓은 화면 활용, 가독성 높은 레이아웃

2. **할일 제목 수정 (인라인 편집)**
   - 행동:
     - "논문 초고 작성" 제목 클릭
     - 텍스트 입력 모드로 전환 (테두리 파란색)
     - "논문 1차 초고 완성" 으로 수정
     - Enter 키 또는 포커스 아웃
   - 서버 요청: PUT /api/todos/:id
   - 예상 결과:
     - 즉시 수정 반영
     - 성공 토스트: "수정되었습니다"

3. **기한 수정**
   - 행동:
     - 할일 우측 "11월 29일" 클릭
     - 날짜 선택기 (date picker) 표시
     - 12월 1일 선택
   - 서버 요청: PUT /api/todos/:id (due_date: 2025-12-01)
   - 예상 결과:
     - 할일 자동 재정렬 (기한 변경으로 순서 변동)
     - 기한 텍스트 업데이트: "12월 1일"

4. **완료된 할일 삭제**
   - 행동:
     - 완료된 "장보기" 할일의 삭제 아이콘 (휴지통) 클릭
     - 확인 대화상자: "정말 삭제하시겠습니까?"
     - "삭제" 버튼 클릭
   - 서버 요청: DELETE /api/todos/:id
   - 예상 결과:
     - 할일 목록에서 제거 (페이드 아웃 애니메이션)
     - "삭제되었습니다" 토스트

5. **키보드 단축키 사용**
   - 행동:
     - 할일 추가 폼에서 Tab 키 (다음 입력란으로 이동)
     - ESC 키 (편집 모드 취소)
   - 예상 결과: 키보드만으로 빠른 조작 가능

**완료 조건**:
- 인라인 편집 성공

### 시나리오 5: 저녁 루틴 - 오늘 완료 및 내일 계획 (모바일)

**배경**:

     - "프로젝트 제출" 체크박스 탭
     - "교수님 미팅 준비" 체크박스 탭
   - 서버 요청: PUT /api/todos/:id (각각)
   - 예상 결과:
     - 체크 애니메이션
     - 취소선 + 회색 처리
     - 목록 하단으로 이동 (완료됨)

2. **내일 할일 추가 (여러 개)**
   - 행동:
     - "+" 버튼 → "강의 준비" (기한: 내일) → 추가
     - "+" 버튼 → "클라이언트 미팅" (기한: 내일 오후) → 추가
     - "+" 버튼 → "장보기" (기한: 내일) → 추가
   - 예상 결과:

     - 주황색 "내일" 배지

3. **완료 목록 확인**

     ✓ 교수님 미팅 준비 (완료, 회색)

   - 예상 결과: 완료된 할일 별도 영역에 표시

4. **성취감 확인**
   - 화면: "오늘 2개 할일을 완료했어요! 🎉" 메시지 (선택)

**완료 조건**:

### 시나리오 12: 동시 수정 충돌 (Concurrent Edit)

**배경**:
민준이 모바일과 데스크톱 두 기기를 동시에 사용 중입니다.

**목표**:

**사전 조건**:
- 모바일과 데스크톱 모두 로그인

**단계**:

   - 기기: iPhone
   - 행동: "논문 읽기" → "논문 정리" 수정
   - 서버 요청: PUT /api/todos/1 (updated_at: 10:00:00)

2. **데스크톱에서 기한 수정 (거의 동시)**
   - 기기: MacBook
   - 행동: 기한 "오늘" → "내일" 수정
   - 서버 요청: PUT /api/todos/1 (updated_at: 10:00:05)

   - 예상 결과:
     - 제목: "논문 읽기" (데스크톱 수정이 덮어씀)
     - 기한: "내일"

4. **모바일 새로고침**
   - 행동: Pull-to-Refresh
   - 예상 결과:
     - 최신 상태 반영
     - 제목 "논문 읽기", 기한 "내일"

**완료 조건**:
- Last Write Wins 정책 적용
- 데이터 일관성 유지

**대체 흐름**:
